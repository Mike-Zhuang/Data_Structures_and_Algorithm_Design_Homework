## 1. 涉及数据结构和相关背景

本次实验主要围绕数组、字符串、排序、分治、堆和双指针展开。几个题目表面上不完全一样：有的是求逆序对数量，有的是把数字拼成最大数，有的是直接比较多种排序算法，还有的是寻找三数之和为 0 的三元组。但做下来以后能感觉到，它们背后有一条共同的线：**先利用某种有序性，再减少不必要的枚举**。

数组是这几题最基础的数据结构。逆序对、排序、三数之和都直接处理整数数组；最大数虽然输入也是整数数组，但真正比较时需要转成字符串。排序在本次实验里既是独立题目，也是其他题目的工具。比如求逆序对用归并排序的合并过程计数，三数之和先排序后用双指针，最大数则用自定义排序规则决定字符串拼接顺序。

分治思想主要体现在归并排序中。它把一个大区间分成左右两半，分别处理后再合并。堆排序用到完全二叉树的数组表示方式，通过父子下标关系维护大根堆。双指针则利用排序后的单调性，让两个指针从两端向中间移动，从而把原本可能需要两重循环枚举的部分压缩成一次线性扫描。

## 2.1 求逆序对

### 2.1.1 问题描述

对于一个长度为 `N` 的整数序列 `A`，如果存在 `i < j` 且 `A[i] > A[j]`，则数对 `(i, j)` 是一个逆序对。题目要求对多组输入分别输出逆序对数量，当输入 `0` 时结束。

样例中，升序序列 `1 2 3 4 5` 的逆序对数量为 `0`；降序序列 `5 4 3 2 1` 中任意两个不同位置都构成逆序对，因此数量为 `10`。

### 2.1.2 基本要求

输入包含多组测试数据，每组先输入整数 `N`，再输入 `N` 个整数。每组数据输出一行逆序对数量。数据范围中 `N <= 20000`，如果直接双重循环比较所有 `(i, j)`，最坏复杂度为 `O(n^2)`，在较大测试点上会超时，因此需要更高效的算法。

### 2.1.3 数据结构设计

本题使用普通数组保存输入序列，并额外使用一个临时数组辅助归并。

```cpp
int group[20005];   // 原始数组，同时在归并过程中被逐步排序
int temp[20005];    // 归并时暂存合并结果
int count;          // 当前测试数据的逆序对数量
```

这里没有使用复杂结构体。关键在于 `group[left..right]` 表示当前正在统计和排序的区间，`temp[left..right]` 用于临时保存合并后的有序区间。

### 2.1.4 功能说明（函数、类）

程序的核心函数有三个：

```cpp
/**
 * @brief 统计数组 group 中的逆序对数量
 * @param group 输入数组
 * @param n     数组长度
 * @param count 用指针返回逆序对数量
 */
void calculate(int group[], int n, int *count);

/**
 * @brief 对区间 [left, right] 做归并排序，并返回该区间逆序对数量
 */
int merge_sort_count(int a[], int temp[], int left, int right);

/**
 * @brief 合并两个有序段 [left, mid] 和 [mid+1, right]，并统计跨区间逆序对
 */
int merge_and_count(int a[], int temp[], int left, int mid, int right);
```

最关键的逻辑在合并阶段。因为左右两段已经各自有序，如果出现 `a[i] > a[j]`，那么左半段从 `i` 到 `mid` 的所有元素都大于 `a[j]`，所以可以一次性加上：

```cpp
inversion += mid - i + 1;
```

这一步是从暴力枚举变快的核心。暴力算法是一个一个数对检查，而归并计数可以在发现一个较小的右半段元素时，批量计算它和左半段剩余元素形成的逆序对。

### 2.1.5 调试分析（遇到的问题和解决方法）

一开始我写的是两层循环，逻辑直观，但是测试结果前几个点是 `WRONG`，后面很多点是 `Time Limit Exceeded`。后来检查发现有几个问题：第一，输出时没有换行，多组结果会黏在一起；第二，代码里曾经出现过类似 `group[n] = {0}` 的写法，而当时 `n` 还没读入，这是不合理的；第三，也是最主要的问题，双重循环复杂度太高。

修正输出格式后，错误点减少，但超时仍然存在。最后把 `calculate` 从暴力计数改成归并排序计数。这个过程里我最容易漏的是 `merge_and_count` 的收尾：主循环结束后还要把左半段或右半段剩余元素复制到 `temp`，最后再把 `temp[left..right]` 拷回 `a[left..right]`。如果不拷回，上一层递归拿到的就不是有序段，结果会错。

### 2.1.6 总结和体会

这题让我比较清楚地体会到，算法复杂度不是最后才考虑的东西。`O(n^2)` 的写法在小样例上完全正确，但一到隐藏测试就会超时。归并排序本来是排序算法，但它的“合并两个有序段”过程刚好能用来统计逆序对，这说明有时候题目的目标不是排序，但排序过程中的结构可以帮助我们解决问题。

## 2.2 最大数

### 2.2.1 问题描述

给定一组非负整数 `nums`，需要重新排列每个数的顺序，使它们拼接后组成最大的整数。由于结果可能很大，不能用普通整数类型保存，最终需要返回字符串。

例如输入 `3 30 34 5 9`，正确输出是 `9534330`。

### 2.2.2 基本要求

输入第一行为数组长度 `n`，第二行为 `n` 个非负整数。输出一行字符串，表示重新排列后能得到的最大整数。数据范围为 `1 <= nums.size() <= 100`，`0 <= nums[i] <= 10^9`。

本题的关键不是按整数大小排序，而是按“两个数拼接后谁放前面更优”排序。

### 2.2.3 数据结构设计

本题主要使用 `vector<int>` 保存输入，排序后逐个转成字符串拼接。

```cpp
vector<int> nums;    // 输入数字
string ans_str;      // 最终拼接结果
string temp_num;     // 当前数字转成的字符串
```

虽然输入是整数，但比较时需要使用字符串拼接。例如比较 `x` 和 `y` 时，不看 `x > y`，而是比较 `to_string(x) + to_string(y)` 与 `to_string(y) + to_string(x)`。

### 2.2.4 功能说明（函数、类）

模板要求提交 `Solution` 类中的 `largestNumber` 函数。

```cpp
/**
 * @brief 将非负整数数组重新排列，返回能拼出的最大数字符串
 * @param nums 输入数组
 * @return 最大拼接结果
 */
string largestNumber(vector<int> &nums);
```

核心比较规则如下：

```cpp
sort(nums.begin(), nums.end(), [](int x, int y) {
    return to_string(x) + to_string(y) > to_string(y) + to_string(x);
});
```

排序后如果第一个元素为 `0`，说明所有元素都是 `0`，需要直接返回 `"0"`，否则会输出 `"000"` 这类不符合要求的结果。

### 2.2.5 调试分析（遇到的问题和解决方法）

一开始我尝试从左到右扫描数组，把当前数字转成字符串后，只判断它应该放到已有答案串的最前面还是最后面。这个方法能过一部分简单测试，但遇到下面这个用例就出错：

```text
nums = [10, 2, 9, 39, 17]
我的输出：92103917
预期输出：93921710
```

问题在于，新来的数字不一定只应该放在最前或最后。比如 `39` 最合适的位置是在 `9` 后面、`2` 前面，而不是整个字符串的两端。也就是说，我当时的算法只比较了很少一部分排列，没有真正决定所有元素之间的相对顺序。

后来改成对任意两个元素 `x`、`y` 比较 `x+y` 和 `y+x`。如果 `x+y` 更大，就让 `x` 排在 `y` 前。这样排序算法会全局调整所有元素的位置，而不是只做局部插入。

### 2.2.6 总结和体会

这题最容易误解的地方是“最大数”不是数字大小降序排序。比如 `9` 应该排在 `39` 前，因为 `939 > 399`；`39` 又应该排在 `2` 前，因为 `392 > 239`。通过这个题我理解到，自定义排序的关键是找到正确的比较规则，而不是硬套普通大小比较。

## 2.3 排序

### 2.3.1 问题描述

本题要求输入 `n` 个整数，输出从小到大排序后的结果。同时实验要求我们用不同排序算法测试不同规模和不同分布的数据，记录运行时间，并在报告中总结各算法的特点、时间复杂度、空间复杂度、稳定性以及对部分有序数据的表现。

### 2.3.2 基本要求

OJ 提交时，只需要实现模板中的 `mySort` 函数，使其返回排序后的数组即可。实验报告部分则需要测试快速排序、归并排序、堆排序、选择排序、冒泡排序、直接插入排序、希尔排序，并比较它们在不同数据规模下的运行时间。

本次实验中，我额外保留了 `std::sort` 作为对照组。测试数据包括随机 `10`、`100`、`1K`、`10K`、`100K`、`1M`，以及 `10K` 正序和 `10K` 逆序。

### 2.3.3 数据结构设计

排序题主要使用 `vector<int>` 保存数组。不同算法使用的辅助结构略有不同：

```cpp
vector<int> nums;     // 输入数组
vector<int> sorted;   // 排序时复制出的数组
vector<int> temp;     // 归并排序使用的临时数组
```

堆排序没有显式建树，而是用数组表示完全二叉树。若父节点下标为 `root`，则：

```cpp
int left_child = root * 2 + 1;
int right_child = left_child + 1;
```

排序实验的计时数据由 `benchmark_sorting.py` 生成，图表由 `plot_sorting_charts.py` 根据 `results.csv` 绘制。

### 2.3.4 功能说明（函数、类）

每个排序文件都按照模板实现：

```cpp
/**
 * @brief 对 nums 从小到大排序
 * @param nums 输入数组
 * @return 排序后的数组
 */
vector<int> mySort(vector<int> &nums);
```

各排序算法的核心思路如下：

| 算法 | 核心思路 |
| --- | --- |
| 快速排序 | 选取基准值，把区间分成不大于基准和不小于基准的两部分，再递归处理 |
| 归并排序 | 分治处理左右区间，再把两个有序段合并 |
| 堆排序 | 建大根堆，每次把堆顶最大值交换到数组末尾 |
| 选择排序 | 每轮从未排序部分选出最小值，放到当前位置 |
| 冒泡排序 | 相邻元素两两比较，大的逐步“冒”到后面 |
| 直接插入排序 | 维护左侧有序区间，把当前元素插入正确位置 |
| 希尔排序 | 按 gap 分组做插入排序，逐步缩小 gap 到 1 |

算法性质总结如下：

| 算法 | 平均时间复杂度 | 最坏时间复杂度 | 空间复杂度 | 稳定性 | 是否适合部分有序数据 |
| --- | --- | --- | --- | --- | --- |
| 快速排序 | `O(n log n)` | `O(n^2)` | `O(log n)` | 不稳定 | 一般，取决于 pivot 选择 |
| 归并排序 | `O(n log n)` | `O(n log n)` | `O(n)` | 稳定 | 表现稳定 |
| 堆排序 | `O(n log n)` | `O(n log n)` | `O(1)` | 不稳定 | 一般 |
| 选择排序 | `O(n^2)` | `O(n^2)` | `O(1)` | 不稳定 | 不明显 |
| 冒泡排序 | `O(n^2)` | `O(n^2)` | `O(1)` | 稳定 | 若加入提前结束优化会较好，本实验版本未重点优化 |
| 直接插入排序 | `O(n^2)` | `O(n^2)` | `O(1)` | 稳定 | 很适合，接近有序时接近 `O(n)` |
| 希尔排序 | 与 gap 序列有关，通常优于 `O(n^2)` | 可能为 `O(n^2)` | `O(1)` | 不稳定 | 较适合中等规模和部分有序数据 |

## 2.4 三数之和

### 2.4.1 问题描述

给定整数数组 `nums`，判断是否存在三个不同下标 `i`、`j`、`k`，使得：

```text
nums[i] + nums[j] + nums[k] == 0
```

题目要求输出所有不重复的三元组，每个三元组内部从小到大排列，所有三元组按最小元素从小到大输出。

### 2.4.2 基本要求

输入两行，第一行为数组长度，第二行为数组元素。数据范围为 `3 <= nums.length <= 3000`，元素范围为 `-10^5` 到 `10^5`。如果用三重循环枚举所有三元组，复杂度是 `O(n^3)`，当 `n=3000` 时不可接受。

### 2.4.3 数据结构设计

本题使用 `vector<int>` 保存输入，使用二维 `vector` 保存答案。

```cpp
vector<int> nums;              // 输入数组，排序后用于双指针扫描
vector<vector<int>> ans;       // 保存所有不重复三元组
```

排序后，三元组内部天然可以按从小到大输出；同时外层从左到右枚举第一个数，也有利于保证答案顺序。

### 2.4.4 功能说明（函数、类）

核心函数如下：

```cpp
/**
 * @brief 返回所有和为 0 且不重复的三元组
 * @param nums 输入数组
 * @return 三元组列表
 */
vector<vector<int>> three_sum(vector<int> &nums);
```

主要流程：

```cpp
sort(nums.begin(), nums.end());

for (int first = 0; first < n; first++) {
    跳过重复的 first;
    int left = first + 1;
    int right = n - 1;

    while (left < right) {
        int sum = nums[first] + nums[left] + nums[right];
        if (sum == 0) {
            记录答案，并跳过重复的 left 和 right;
        } else if (sum < 0) {
            left++;
        } else {
            right--;
        }
    }
}
```

排序后，如果 `sum < 0`，说明当前和太小，需要让数变大，所以移动 `left`；如果 `sum > 0`，说明当前和太大，需要让数变小，所以移动 `right`。

### 2.4.5 调试分析（遇到的问题和解决方法）

我一开始对复杂度有疑问：外层固定第一个数，内层又有 `left` 和 `right` 两个变量在动，看起来像三层。但后来理清楚了，双指针不是两层循环。对于某个固定的 `first`，`left` 只会向右走，`right` 只会向左走，每次循环至少有一个指针移动，两个指针总共最多扫描一遍剩余区间。因此内层是 `O(n)`，外层是 `O(n)`，总复杂度是 `O(n^2)`。

另一个需要注意的问题是去重。外层 `first` 如果和前一个值相同，需要跳过；找到一个答案后，`left` 和 `right` 也要跳过重复值，否则会输出重复三元组。例如样例中有两个 `-1`，但 `[-1, 0, 1]` 只能输出一次。

### 2.4.6 总结和体会

这题把排序和双指针结合得很典型。排序的目的不是为了直接输出，而是为了让数组具有单调性，从而可以根据当前和的大小决定指针移动方向。相比三重循环，双指针减少了大量重复枚举。这个题也提醒我，看到多个变量变化时不能只数变量个数判断复杂度，而要看它们是否嵌套、是否会反复扫描同一段。

## 3. 实验总结

这次 HW5 做下来，我最大的感受是：算法题不能只停留在“样例能过”。逆序对一开始暴力写法很自然，但隐藏测试很快暴露出 TLE；最大数一开始的两端插入思路看起来也有点道理，但遇到 `39` 这种需要插在中间的情况就不行；三数之和如果直接三重循环，数据范围也不允许。很多问题真正的突破点都在于找到更好的结构，比如归并过程、自定义排序规则、堆、双指针。

排序实验也让我对复杂度有了更直观的认识。以前看到 `O(n^2)` 和 `O(n log n)` 只是符号，这次实际跑到 `1M` 数据以后，选择排序和冒泡排序需要几百秒，而快速排序、归并排序、堆排序、希尔排序和 `std::sort` 只需要一两秒左右。这个差距不是微调代码能弥补的，而是算法级别的差距。

另外，我觉得这次实验里“调试过程”也很重要。比如输出少一个换行会导致 Wrong Answer，归并时忘记拷回原数组会导致上层递归错误，最大数忘记全 0 特判会输出不规范结果。这些问题不一定很高级，但都是真正写代码时会遇到的细节。

总的来说，这次作业让我更熟悉了数组、排序、分治、堆和双指针这些基础方法，也让我更愿意在写代码前先根据数据范围判断复杂度。后面再遇到类似题目时，我会先想清楚“暴力能不能过”“有没有排序后可利用的有序性”“有没有重复结果需要去重”，而不是直接开始写循环。
